\section{Experimental Setup}
\label{sec:eval-setup}

We evaluate on SWE-bench Lite~\citep{jimenez2023swebench}, a curated subset of 300 instances drawn from 11 open-source Python repositories:

\begin{center}
\begin{tabular}{ll}
\texttt{astropy}            & \texttt{mwaskom/seaborn}   \\
\texttt{django}             & \texttt{psf/requests}      \\
\texttt{flask}              & \texttt{pydata/xarray}     \\
\texttt{matplotlib}         & \texttt{pylint-dev/pylint} \\
\texttt{pallets/flask}      & \texttt{pytest-dev/pytest} \\
\texttt{scikit-learn}       &                            \\
\end{tabular}
\end{center}

Each instance corresponds to a GitHub issue with a failing test that was introduced by a real commit. The benchmark defines a binary resolution criterion: an instance is \emph{resolved} if the submitted patch causes all originally failing tests to pass without breaking any previously passing tests.

Each instance runs inside an isolated Docker container managed by \texttt{TracedInstanceRunner}, as described in \Cref{ch:system-methodology}. We run each instance under two \emph{variants}:

\begin{itemize}\setlength{\itemsep}{6pt}
    \item \textbf{without\_runtime}: the LLM session uses only the five static tools (\texttt{print\_project\_tree}, \texttt{open\_file}, \texttt{search\_symbol}, \texttt{get\_test\_error}, \texttt{apply\_patch}). This serves as the baseline.
    \item \textbf{with\_runtime}: the full nine-tool catalog is available, including the runtime tools (\texttt{get\_execution\_trace}, \texttt{get\_frames}, \texttt{get\_granular\_frames}). The tracer runs before the LLM session and populates \texttt{auto\_debug.json}.
\end{itemize}

We evaluate both variants across the following base models:

\begin{itemize}\setlength{\itemsep}{6pt}
    \item Frontier models: \texttt{DeepSeek-V4-Pro}, \texttt{gpt-5.4-mini}, \texttt{claude-sonnet-4.6}
    \item Small open-source models: \texttt{Qwen2.5-Coder-7B-Instruct}
\end{itemize}

We deliberately evaluate on both large frontier models and small open-source models. This allows us to assess whether the benefit of runtime information depends on model scale. Our hypothesis is that runtime data yields a larger relative improvement for smaller models, as there is greater room to compensate for their lack of world knowledge.

All models are accessed via OpenAI-compatible APIs. The default number of tool runs in session runs is 10.

We run a single LLM session per instance per variant. We do not ensemble or run multiple trials. We set temperature to 0 where the API allows it to make runs deterministic, though not all providers guarantee identical outputs at temperature 0. We report results on the full 300-instance set unless otherwise noted.

\section{Evaluation Metrics}
\label{sec:eval-metrics}

Let $\mathcal{I}$ be the set of benchmark instances, $\mathcal{M}$ a base model, and $v \in \{\texttt{with\_runtime},\, \texttt{without\_runtime}\}$ a variant. Each run produces a status $s(i, \mathcal{M}, v) \in \{\texttt{passed}, \texttt{failed}, \texttt{apply\_failed}\}$ for each instance $i \in \mathcal{I}$.

\paragraph{Resolved rate.} The primary metric is:
\[
  R(\mathcal{M}, v) \;=\; \frac{\bigl|\{\, i \in \mathcal{I} \;\mid\; s(i, \mathcal{M}, v) = \texttt{passed} \,\}\bigr|}{|\mathcal{I}|}
\]
This matches the standard SWE-bench \% resolved metric and allows direct comparison with published results.

\paragraph{Localization accuracy.} The gold patch for each instance specifies which files, functions, and lines were modified to fix the bug. We compare the patch produced by the model against the gold patch and compute accuracy at three granularities:
\[
  L_g(\mathcal{M}, v) \;=\; \frac{|\{\, i \in \mathcal{I} \;\mid\; \mathrm{loc}_g(i, \mathcal{M}, v) = 1 \,\}|}{|\mathcal{I}|}, \quad g \in \{\texttt{file},\, \texttt{func},\, \texttt{line}\}
\]
where $\mathrm{loc}_g(i, \mathcal{M}, v) = 1$ if the model's patch touches at least one of the correct files (resp. functions, lines) identified in the gold patch. File-level localization is the weakest criterion and line-level the strictest.

\paragraph{Token count.} We report the mean input and output token counts per instance:
\[
  T_{\mathrm{in}}(\mathcal{M}, v) \;=\; \frac{1}{|\mathcal{I}|}\sum_i n_{\mathrm{in}}(i, \mathcal{M}, v), \qquad
  T_{\mathrm{out}}(\mathcal{M}, v) \;=\; \frac{1}{|\mathcal{I}|}\sum_i n_{\mathrm{out}}(i, \mathcal{M}, v)
\]
Token counts are a model-independent efficiency signal. Unlike cost, they do not depend on per-provider pricing, so they remain comparable across providers and over time as prices change. Comparing $T_{\mathrm{in}}$ across variants reveals how much context the runtime tools add to the prompt. Comparing $T_{\mathrm{out}}$ shows whether the model produces longer or shorter responses when runtime data is available.

\section{Ablation Studies (RQ2)}
\label{sec:eval-rq2}

RQ2 asks which runtime data structures drive the improvement. The full runtime tool set exposes three data structures: the execution path (\texttt{exec\_path}), the return-event frame snapshots (\texttt{frames}), and the line-level step frames (\texttt{step\_frames}). We ablate these by restricting the tool catalog to each structure in isolation and measuring the effect on resolved rate and localization.

\begin{table}[h]
\centering
\caption{Ablation of runtime data structures on SWE-bench Lite (\texttt{gpt-5.4-mini}). All values are dummy placeholders pending full evaluation.}
\label{tab:ablation}
{\small
\begin{tabular}{lcccc}
\toprule
Configuration & Resolved\% & File\% & Func\% & Line\% \\
\midrule
\texttt{no\_runtime} (baseline)                               & 28.0 & 72.0 & 55.0 & 42.0 \\
\texttt{exec\_path} only                                      & 29.0 & 73.0 & 56.0 & 43.0 \\
\texttt{frames} only                                          & 30.0 & 74.0 & 57.0 & 44.0 \\
\texttt{step\_frames} only                                    & 31.0 & 75.0 & 58.0 & 45.0 \\
\texttt{exec\_path} + \texttt{frames}                         & 31.0 & 75.0 & 58.0 & 45.0 \\
\texttt{exec\_path} + \texttt{step\_frames}                   & 32.0 & 76.0 & 59.0 & 46.0 \\
\texttt{frames} + \texttt{step\_frames}                       & 32.0 & 76.0 & 60.0 & 47.0 \\
\texttt{exec\_path} + \texttt{frames} + \texttt{step\_frames} & \textbf{33.0} & \textbf{76.0} & \textbf{60.0} & \textbf{47.0} \\
\bottomrule
\end{tabular}
}
\end{table}

Each row in \Cref{tab:ablation} tests a distinct hypothesis about which data type carries diagnostic value. \texttt{exec\_path} only tests whether the call graph alone, without variable state, is sufficient to improve resolution. \texttt{frames} only tests whether return-event variable snapshots suffice for the model to locate and repair a bug. \texttt{step\_frames} only tests whether line-level granularity adds value over return frames. The combined configurations reveal whether the data types interact: gains from pairing two structures may exceed or fall short of what either provides in isolation.

We expect \texttt{step\_frames} to contribute most to the resolved rate. Step frames expose variable state at every source line and allow the model to pinpoint exactly where a value becomes incorrect, which is the information most directly useful for patch generation. We expect \texttt{exec\_path} alone to provide moderate localization improvement, because the call graph narrows the function space the model must search, but limited repair improvement, because it carries no variable state. If the full combination of all three structures significantly outperforms any single structure, it suggests the data types are complementary and that path coverage, return-event snapshots, and line-level state each contribute information the others do not.
