\section{Experimental Setup}
\label{sec:eval-setup}

Our primary evaluation dataset is SWE-bench Lite~\citep{jimenez2023swebench}: a collection of 300 real-world bugs and their fixed, collected from 11 open-source Python repositories on GitHub:

\begin{center}
\begin{tabular}{ll}
\texttt{astropy}            & \texttt{mwaskom/seaborn}   \\
\texttt{django}             & \texttt{psf/requests}      \\
\texttt{flask}              & \texttt{pydata/xarray}     \\
\texttt{matplotlib}         & \texttt{pylint-dev/pylint} \\
\texttt{pallets/flask}      & \texttt{pytest-dev/pytest} \\
\texttt{scikit-learn}       &                            \\
\end{tabular}
\end{center}

Each data sample corresponds to \xxx{TODO define data format}. The ground truth is a reference patch and a data sample is considered resolved, if the produced patch makes tests pass without breaking any previously passing tests.

Each instance runs inside an isolated Docker container managed by \texttt{TracedInstanceRunner}, as described in \Cref{ch:system-methodology}. We run each instance under two variants (modes):

\begin{itemize}\setlength{\itemsep}{6pt}
    \item \texttt{without\_runtime}: the model is allowed to only use static context via static tools (\texttt{print\_project\_tree}, \texttt{open\_file}, \texttt{search\_symbol}, \texttt{get\_test\_error}, \texttt{apply\_patch}). This serves as the baseline.
    \item \texttt{with\_runtime}: the model is allowed to use all the available tools, including the runtime tools (\texttt{get\_execution\_trace}, \texttt{get\_frames}, \texttt{get\_granular\_frames}).
\end{itemize}

We evaluate both variants across the following base models:

\begin{itemize}\setlength{\itemsep}{6pt}
    \item Frontier models: \texttt{DeepSeek-V4-Pro}, \texttt{gpt-5.4-mini}, \texttt{claude-sonnet-4.6}
    \item Small open-source models: \texttt{Qwen2.5-Coder-7B-Instruct}
\end{itemize}

We deliberately evaluate on both large frontier models and small open-source models. This allows us to assess whether the benefit of runtime information depends on model scale. Our hypothesis is that runtime data yields a larger relative improvement for smaller models, as there is greater room to compensate for their lack of world knowledge.

All models are accessed via OpenAI-compatible APIs. The default number of tool runs in session runs is 10.

We run a single LLM session per instance per variant (mode). We do not ensemble or run multiple trials. We set temperature to 0 where the API allows it to make runs deterministic, though not all providers guarantee identical outputs at temperature 0. We report results on all 300 dataset samples unless otherwise noted.

\section{Evaluation Metrics}
\label{sec:eval-metrics}

Let $\mathcal{I}$ be the set of benchmark instances, $\mathcal{M}$ a base model, and $v \in \{\texttt{with\_runtime},\, \texttt{without\_runtime}\}$ a variant. Each run produces a status $s(i, \mathcal{M}, v) \in \{\texttt{passed}, \texttt{failed}, \texttt{apply\_failed}\}$ for each instance $i \in \mathcal{I}$.

\paragraph{Resolved rate.} The primary metric is:
\[
  R(\mathcal{M}, v) \;=\; \frac{\bigl|\{\, i \in \mathcal{I} \;\mid\; s(i, \mathcal{M}, v) = \texttt{passed} \,\}\bigr|}{|\mathcal{I}|}
\]
This matches the standard SWE-bench \% resolved metric and allows direct comparison with published results.

\paragraph{Localization accuracy.} The gold patch for each instance specifies which files, functions, and lines were modified to fix the bug. We compare the patch produced by the model against the gold patch and compute accuracy at three granularities:
\[
  L_g(\mathcal{M}, v) \;=\; \frac{|\{\, i \in \mathcal{I} \;\mid\; \mathrm{loc}_g(i, \mathcal{M}, v) = 1 \,\}|}{|\mathcal{I}|}, \quad g \in \{\texttt{file},\, \texttt{func},\, \texttt{line}\}
\]
where $\mathrm{loc}_g(i, \mathcal{M}, v) = 1$ if the model's patch touches at least one of the correct files (resp. functions, lines) identified in the gold patch. File-level localization is the weakest criterion and line-level the strictest.

\paragraph{Token count.} We report the mean input and output token counts per instance:
\[
  T_{\mathrm{in}}(\mathcal{M}, v) \;=\; \frac{1}{|\mathcal{I}|}\sum_i n_{\mathrm{in}}(i, \mathcal{M}, v), \qquad
  T_{\mathrm{out}}(\mathcal{M}, v) \;=\; \frac{1}{|\mathcal{I}|}\sum_i n_{\mathrm{out}}(i, \mathcal{M}, v)
\]
We decided to include token counts as an efficiency signal, so we can compare the usage across different models. Comparing $T_{\mathrm{in}}$ across variants reveals how much context the runtime tools add to the prompt. Comparing $T_{\mathrm{out}}$ shows whether the model produces longer or shorter responses when runtime data is available.
