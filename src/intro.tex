
\chapter{Introduction}

\section{Motivation}
Agent-based systems now represent the standard approach to LLM-based
automated program repair. \citet{yang2024sweagent} introduced SWE-agent,
which gives the model a custom tool interface for file inspection, code
search, and test execution. \citet{zhang2024autocoderover} proposed
AutoCodeRover, which replaces raw file reads with AST-level retrieval and
spectrum-based fault localization. \citet{xia2024agentless} designed
Agentless, a fixed pipeline of hierarchical localization, patch sampling,
and test-based candidate selection. On SWE-bench Lite
\citep{jimenez2023swebench}, these three systems reached 12.5\%, 19\%,
and 32\% pass@1, respectively.

The repair process these systems follow has no analog in how developers actually debug.
A developer investigating a test failure reads the call stack, inspects argument values
at each frame, and checks variable states at the crash point, typically through a
debugger or execution trace. These systems give the model none of that. They supply
source code and a test failure message, and the model must reconstruct what the program
did from text alone. \citet{chen2023teaching} point to this as the core constraint of
execution-free repair, and \citet{zhong2024debug} show that explicitly providing models
with control flow and variable state information before patch generation raises
correctness.

This thesis asks whether supplying runtime information collected during
test execution improves patch correctness. We instrument Python unit
tests to collect call stacks, argument values, and exception state, and
we supply the collected data in the LLM prompt alongside the failing test
and source code. We evaluate this approach against a static-context
baseline on real Python bugs and build on prior evidence that
execution-derived context improves model reasoning on code tasks
\citep{ni2024nextteachinglargelanguage,han2025rethinkingcapabilityfinetunedlanguage}.

\section{Our contributions}
We present a system for runtime-augmented LLM-based program repair.
The specific contributions are:
\begin{enumerate}
  \item \textbf{Execution tracer for failing unit tests.} We instrument Python test suites via
    \texttt{sys.settrace} to collect, per failing test, the call stack, executed function names,
    and argument values at function entry. We bound collection depth and serialize records into
    structured form.
  \item \textbf{Multi-turn tool interface for repair.} We design a session in which the LLM
    accesses runtime data through tool calls: file reading, call-graph queries, argument
    inspection, and test execution. The model selects which data to retrieve and when.
  \item \textbf{Comparative and ablative experiments.} On SWE-bench Lite and BugsInPy, we
    measure patch correctness for a no-context baseline, static trace injection, and the
    interactive interface. Ablations over trace subsets quantify how much each category of
    runtime information contributes to repair success.
  \item \textbf{Fine-tuning on runtime-augmented data.} We fine-tune Qwen2.5-Coder-7B-Instruct on approximately 300 training examples constructed from the benchmark traces, pairing each initial issue prompt and its runtime trace with the gold reference patch. We train with QLoRA on a single A100 GPU across five seeds and evaluate whether fine-tuning on task-specific runtime data further improves repair over prompting alone.
\end{enumerate}
