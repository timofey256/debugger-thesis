
\chapter{Introduction}

\section{Motivation}
Programming assistants based on large language models (LLMs) have become widely used for tasks such as code completion, refactoring, and bug fixing. In the context of automated program repair (APR), current LLM-based approaches are usually prompted with a static snapshot of the source code together with a failing unit test and the failure output, and they are expected to propose a patch that makes the test pass \citep{yang2025surveyllmbasedautomatedprogram,zhang2023surveylearningbasedautomatedprogram}.

A key limitation of this setting is that test failures are caused by a specific runtime behavior: certain functions are executed, particular values flow through variables, and an exception (or wrong output) is produced at a concrete program point. If the model only sees static code, it must infer this runtime state indirectly, which is difficult in general and particularly challenging in dynamically typed languages such as Python. For example, the model may not know which branch was actually taken, what the true types or shapes of values were, or what arguments were passed to a misbehaving function.

The aim of this thesis is to investigate whether supplying the LLM with program information collected during unit test execution can improve its ability to produce correct fixes for failing tests. More broadly, the thesis is motivated by recent work arguing that LLMs benefit from being guided by information about program execution, rather than relying solely on static reasoning over source code \citep{ni2024nextteachinglargelanguage, han2025rethinkingcapabilityfinetunedlanguage}.

\section{Our contributions}
This thesis makes the following contributions:
\begin{enumerate}
  \item \textbf{Runtime context collection for tests.} We implement an instrumentation pipeline for Python unit tests that collects execution-derived information such as executed methods/functions and observed argument values (in a bounded, serializable form).
  \item \textbf{Runtime-augmented prompting for repair.} We design prompting templates that incorporate the collected runtime context when asking an LLM to fix a failing unit test, and we integrate patch validation by re-running tests.
  \item \textbf{Experimental evaluation.} We evaluate the proposed system against a baseline that uses no runtime information, and we analyze the impact of different kinds and amounts of runtime context; where applicable, we also compare multiple models and explore fine-tuning as an additional approach.
\end{enumerate}

\end{tcolorbox}

The remainder of the thesis is organized as follows. \Cref{ch:related-work} reviews related work on learning-based program repair, LLM-based test repair, and execution-aware approaches. \Cref{ch:methodology} presents our approach, including runtime information collection, prompt construction, and implementation details. \Cref{ch:evaluation} describes the experimental setup, datasets, metrics, and results, including ablations over runtime context. \Cref{ch:discussion} discusses limitations and threats to validity. Finally, \Cref{ch:conclusion} summarizes the findings and outlines future work.
