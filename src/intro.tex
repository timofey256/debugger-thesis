\section{Motivation}
Automated program repair (APR) is the task of locating and fixing bugs in code.
Success is typically defined as whether the change patches a broken behaviour. 
Early APR approaches relied on utilisation of static context, such as abstract syntax trees (ASTs), predefined fix templates, 
search-based methods that mutated code until tests passed, or constraint solvers that synthesized repairs from formal specifications.
With the rise of large language models, the field shifted toward the repair based non-deterministic models that are trained on large code and text corpora.
Recent surveys show that agent-based repair becomes a dominant paradigm~\citep{zhang2024slrapr, liu2024agentsse}.
We define agents as loops of LLM forward passes supplied with the available context. Agents often interact with the environment through
requesting files content, search for symbols, and run tests. This loop iterates until a fixing patch is produced.

Most of current agent systems share a common assumption that static context is enough for repair.
For example, \citet{zhang2024autocoderover} proposed
AutoCodeRover, which replaces raw file reads with AST-level retrieval and
spectrum-based fault localization. \citet{xia2024agentless} designed
Agentless, a fixed pipeline of hierarchical localization, patch sampling,
and test-based candidate selection. On SWE-bench Lite
\citep{jimenez2023swebench}, these three systems reached 12.5\%, 19\%,
and 32\% pass@1, respectively.

However, a fundamental limitation of this approach is that it's not how humans actually debug programs.
When investigating a failed test, developer can read the call stack, inspect how variables
change over time, and check the program state at the crash point. Execution-free systems give the model none of that.
They provide the source code and an exception message, and the model must reconstruct what the program
did from static context alone. \citet{chen2023teaching} point to this as the core constraint of
execution-free repair, and \citet{zhong2024debug} show that explicitly providing models
with control flow and variable state information before patch generation boosts performance.

\section{Our contributions}
\begin{enumerate}
  \item Custom debugger hooks for trace collection. We developed a suite of custom debugger hooks that
      allow collecting runtime traces in the most common Python test frameworks: \texttt{pytest}, \texttt{unittest}, \texttt{django} (modification of \texttt{unittest}).
      The collected traces are outputed in a structured JSON format and contain entry/exit frames and line-by-line frames with local variables.
  \item Comparative and ablative experiments. On SWE-bench Lite and BugsInPy, we
    measure patch correctness for a no-context baseline, static trace injection, and the
    interactive interface. Ablations over trace subsets quantify how much each category of
    runtime information contributes to repair success.
  \item Fine-tuning on runtime-augmented data. We fine-tune Qwen2.5-Coder-7B-Instruct on approximately 300 training samples
      collected using our abovementioned trace-collecting tool. We train with QLoRA on a single A100 GPU across five seeds and
      evaluate whether fine-tuning on task-specific runtime data further improves repair compared to prompting alone.
\end{enumerate}
